Retrieval-Augmented Generation (RAG) is a breakthrough architectural pattern in modern natural language processing. First introduced by Patrick Lewis and colleagues from Facebook AI Research (FAIR) in their landmark 2020 paper titled "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks", RAG combines the strengths of pre-trained parametric language models with external non-parametric knowledge bases. The primary objective of RAG is to solve the inherent limitations of Large Language Models (LLMs), which include factual hallucinations, outdated information, and an inability to access private or proprietary data sources.

The classic RAG architecture operates in three distinct phases: ingestion, retrieval, and generation. During the ingestion phase, documents are extracted, parsed, chunked, and embedded into a vector space before being stored in a specialized database known as a vector store or vector database. Common vector databases include ChromaDB, Pinecone, Milvus, and Qdrant. During the retrieval phase, when a user submits a query, the system converts this query into an embedding vector and searches the vector store for the top-N most semantically similar text chunks using distance metrics such as Cosine Similarity, Euclidean Distance, or Dot Product. In the final generation phase, the retrieved text chunks are prepended as context to the user's original query, forming a prompt that is sent to the LLM (the Generator) to produce a grounded, factually accurate response.

A critical, yet often overlooked, component of the ingestion pipeline is text chunking. Chunking is the process of breaking down a large, continuous document into smaller, manageable segments. If chunks are too large, the retrieved context may exceed the context window of the LLM or introduce irrelevant information, diluting the signal-to-noise ratio. Conversely, if chunks are too small, they may lose vital contextual information, rendering the retrieved snippet useless. There are four primary chunking strategies used in production RAG systems today:

1. Fixed-Size Chunking: This is the simplest and most common method. It divides the document into chunks of a predefined number of characters, words, or tokens, with a fixed overlap between adjacent chunks. For example, a system might use a chunk size of 500 characters with a 100-character overlap. While computationally efficient, fixed-size chunking frequently cuts off sentences in the middle of a thought, breaking semantic continuity.

2. Sliding Window Chunking: A token-based variation of fixed-size chunking where the sliding window moves across the text with a set step size. If a window is set to 200 words and the step size is 50 words, the overlap is 150 words. This ensures that every piece of text is represented in multiple overlapping contexts, which increases the likelihood of capturing full sentences and local relationships, though at the expense of storage redundancy.

3. Semantic Chunking: Instead of relying on arbitrary character or token counts, semantic chunking uses embedding models to identify natural transitions in the text. The document is split into individual sentences. The system calculates vector embeddings for each sentence and computes the cosine similarity between consecutive sentences. A split boundary is created wherever the similarity drops below a specified threshold, indicating a change in topic. This results in chunks that are semantically self-contained and highly cohesive.

4. Hierarchical Chunking (Parent-Child): This advanced strategy creates a multi-layered structure of chunks. The document is first split into large "parent" chunks (e.g., 1000 characters), which are then subdivided into smaller "child" chunks (e.g., 200 characters) with overlap. During retrieval, the vector database index searches and matches against the smaller child chunks, which are easier to retrieve accurately. However, instead of passing the tiny child chunk to the LLM, the system retrieves and inserts the larger parent chunk. This provides the LLM with a rich, surrounding context while preserving high-precision search.

Despite the benefits of vector retrieval, first-pass retrieval using cosine similarity on embeddings has notable limitations. Vector embeddings excel at capturing broad semantic concepts but often fail with precise keyword matching, complex queries, or fine-grained rankings. To solve this, developers employ a second-pass retrieval step known as Reranking.

Reranking uses a cross-encoder model, such as Cohere Rerank, to re-evaluate the relevance of the retrieved documents. In first-pass retrieval, the query and document embeddings are calculated independently (bi-encoder architecture) and compared via simple vector math. This is fast but loses interactive nuances. In second-pass reranking, the cross-encoder model processes the query and each document chunk together, allowing self-attention layers to compute deep, query-document interactions. 

For instance, if first-pass retrieval searches a vector store containing 10,000 chunks and returns the top 10 candidates, the Cohere Rerank API (specifically the rerank-english-v3.0 model) will analyze these 10 chunks alongside the query and output a relevance score between 0.0 and 1.0 for each. The chunks are then re-sorted by their rerank score, and only the top 3 are sent to the LLM. This second-pass step drastically improves retrieval precision, filters out irrelevant background noise, and reduces the LLM's prompt token count, saving money and reducing latency.

To systematically evaluate retrieval quality, RAG developers track key metrics. The two most common retrieval metrics are Hit Rate (or Recall) and Mean Reciprocal Rank (MRR). Hit Rate @ K measures the percentage of queries for which the correct ground-truth chunk is successfully retrieved within the top K results. For example, a Hit Rate @ 3 of 80% means the system found the correct answer in the top 3 results for 8 out of 10 questions. Mean Reciprocal Rank (MRR) @ K evaluates the position of the first correct chunk. If the correct chunk is retrieved at rank 1, the reciprocal rank is 1.0; if at rank 2, it is 0.5; if at rank 3, it is 0.33; and if it is not in the top K, the reciprocal rank is 0.0. The final MRR is the average of these reciprocal ranks across all test queries. A high MRR indicates that the system is not only finding the correct information but placing it at the very top of the results, maximizing LLM generation quality.
