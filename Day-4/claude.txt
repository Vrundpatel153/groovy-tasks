Why It Happens
The problematic code creates a cycle:

Component renders → count = 0
Effect runs → calls setCount(1)
State updates → component re-renders
Dependency array detects count changed
Effect runs again → calls setCount(2)
Back to step 3... ∞
The Three Best Solutions
Solution 1: Empty Dependency Array — Use when you want initialization code to run once on mount

javascript
useEffect(() => {
  setCount(1);
}, []);  // Runs once, never again
Solution 2: Functional Update Form — Use when updating state based on previous state

javascript
useEffect(() => {
  setCount(prev => prev + 1);
}, []);  // prevState doesn't go in dependencies
Solution 3: Update Different State — Use when reacting to one state by updating another

javascript
const [count, setCount] = useState(0);
const [doubled, setDoubled] = useState(0);

useEffect(() => {
  setDoubled(count * 2);  // Update doubled, not count
}, [count]);  // Safe!

The Analogy: From Spreadsheets to Databases
Since you know spreadsheets, imagine them as the starting point. A spreadsheet is basically SQL.

SQL databases are like Excel spreadsheets on steroids:

Think of SQL as a super-organized filing system. Every table (like a sheet) has columns with strict names and types — Name, Age, Email. If you add a new person, they must fit into those exact columns. No variations. It's rigid, predictable, and perfect for things like a company payroll system where everyone has the same information (salary, start date, department). The database enforces the rules — you can't skip a required column.

NoSQL databases are like a folder of documents:

NoSQL is more like having a folder full of sticky notes or text files. Each "document" (a record) can have whatever information you want. One note might say "Name: Alice, Hobby: Tennis, Age: 28." Another might say "Name: Bob, Job Title: Manager, Languages Spoken: English, Spanish." They don't have to match. If Alice doesn't have a job title, that's fine — just leave it out. This flexibility is great when you're storing things that naturally vary, like social media posts (some have photos, some have videos, some are just text).

Key Differences at a Glance
Aspect	SQL	NoSQL
Structure	Rigid schema (fixed columns)	Flexible (any fields)
Scaling	Vertical (bigger server)	Horizontal (more servers)
Relationships	Tables join together	Documents stand alone
Speed	Slower for huge amounts of data	Faster for massive scale
Learning curve	Complex queries (SQL language)	Simpler (often JSON-like)
Real-World Example: Social Media
If you were building a user profile system with SQL:

Every user must have: ID, username, email, phone, bio, date_joined
If you want to add "favorite_color", you have to change the table for everyone
This is slow but very organized
If you were using NoSQL:

User 1: {username: "alice", email: "alice@email.com", hobbies: ["tennis", "reading"]}
User 2: {username: "bob", email: "bob@email.com", followers_count: 1500}
User 3: {username: "carol", email: "carol@email.com", hobbies: ["gaming"], premium: true}
Each user can have completely different fields. Super flexible!
The Catch
SQL requires you to decide your structure upfront. Once you've built the table, changing it is annoying. But this predictability makes it perfect for mission-critical data (banks, hospitals, governments).

NoSQL lets you throw any data at it and figure out the structure later. Perfect for startups that are still figuring out what they need. But too much flexibility can become chaotic — without rules, it's easy to end up with messy, inconsistent data.

A junior developer's rule of thumb: Start with SQL if you're building something structured (customer orders, inventory, finances). Use NoSQL if you're building something flexible and scalable (social feeds, real-time chat, content management). Many modern systems use both — SQL for the important stuff, NoSQL for the flexible stuff.

The Problems with the Original

My Top Pick: The Practical Solution
For most real-world projects, I recommend this:

javascript
function getAverage(numbers) {
  // Guard clauses: check early and return if invalid
  if (!Array.isArray(numbers) || numbers.length === 0) {
    return null;
  }

  // Filter to keep only valid numbers
  const validNumbers = numbers.filter(
    item => typeof item === 'number' && !Number.isNaN(item)
  );

  if (validNumbers.length === 0) {
    return null;
  }

  // Use reduce for a cleaner sum calculation
  const sum = validNumbers.reduce((acc, num) => acc + num, 0);
  return sum / validNumbers.length;
}
Why this approach:
