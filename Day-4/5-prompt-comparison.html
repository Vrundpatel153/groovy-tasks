1) Explain SQL vs NoSQL to a junior developer using a spreadsheet analogy

Database comparison 
Welcome to the world of databases! Since you are already comfortable with spreadsheets like Excel or Google Sheets, you actually have a massive head start. Both SQL and NoSQL databases exist to do exactly what spreadsheets do: store, organize, and retrieve data.

The main difference lies in how they structure that data and how flexible they are. Let’s break it down using a real-world analogy you'd encounter at a local mall.

The Analogy: A Clothing Store vs. A Flea Market
Imagine you are managing inventory for a retail business.

1. SQL is a High-End Clothing Store (Relational)
In a clothing store, everything is meticulously organized into strict sections.

Every item goes into a specific rack.

Every shirt must have a size, a color, a price, and a brand. If a shirt is missing a price tag, the system errors out—you aren't allowed to put it on the floor.

If you want to find a matching outfit, you look at the shirt's tag, which points you directly to the matching pants aisle.

2. NoSQL is a Vibrant Flea Market (Non-Relational)
A flea market is incredibly flexible. You have different vendors selling wildly different things in the same space.

Vendor A is selling vintage leather jackets (which have sizes and years).

Vendor B is selling old comic books (which have issue numbers and conditions, but no "size").

Vendor C is selling handmade jewelry (with no brand or specific category).

If a new vendor shows up tomorrow selling absolute randomness, the flea market doesn't crash. It just pulls up another table and adapts instantly.

How They Map to Databases
SQL (Structured Query Language)
SQL databases are exactly like spreadsheets. If you've ever built a spreadsheet where you locked the column headers and insisted that the "Age" column only contain numbers, you’ve essentially designed a SQL database.

The Structure: Data is stored in Tables (like individual sheets). Tables have strict Columns (fields) and Rows (records).

The Rules (Schema): You must define your structure before you add data. If you have columns for First_Name, Last_Name, and Email, you cannot suddenly add a row that contains a list of a user's favorite movies unless you alter the entire sheet to add a Favorite_Movies column for everyone.

Relationships: SQL excels at linking tables together. For example, a Customers sheet can link to an Orders sheet using a unique ID.

Examples: PostgreSQL, MySQL, SQLite.

NoSQL (Not Only SQL)
NoSQL databases throw out the traditional spreadsheet grid. Instead of rows and columns, the most common type of NoSQL (Document stores) stores data as independent packets, usually looking like standard JSON objects.

The Structure: Data is stored in Documents collected together in folders called Collections.

The Flexibility (Schemaless): Every single document can be completely different. One customer document might just have a name and phone number. The next customer document might have a name, three email addresses, a shipping history, and a list of pet names. NoSQL does not care; it accepts the data as-is.

Examples: MongoDB, DynamoDB, Redis.

Quick Comparison Table
Feature	SQL (The Clothing Store)	NoSQL (The Flea Market)
Data Format	Organized Tables (Rows & Columns)	Flexible Documents, Key-Values, or Graphs
Schema	Strict. Must define columns in advance.	Dynamic. Add any data on the fly.
Relationships	Excellent. Perfect for complex links between data.	Poor. Data is usually self-contained in one place.
Scaling	Scaled by making the server stronger (Vertical).	Scaled by splitting data across more servers (Horizontal).
When should you use which?
Choose SQL if: You are building something where data integrity is life-or-death, like a banking app, a checkout system, or an app where data is highly interconnected (like a social media network's user connections). You want structure, and you want to prevent bad data from getting in.

Choose NoSQL if: You are dealing with rapidly changing data features, unstructured data (like logging user behavior, IoT sensor data, or content management systems), or you need to scale to massive amounts of data incredibly fast without worrying about strict table relationships.

JavaScript
/**
 * Calculates the average of an array of numbers.
 * @param {Array} arr - The array of numbers.
 * @returns {number} The average value, or 0 if the array is empty or invalid.
 */
function avg(arr) {
  // 1. Handle edge case: Check if input is a valid, non-empty array
  if (!Array.isArray(arr) || arr.length === 0) {
    return 0;
  }

  // 2. Filter out non-number values and handle invalid number types like NaN
  const validNumbers = arr.filter(num => typeof num === 'number' && !isNaN(num));

  // 3. Handle edge case: Check if any valid numbers remain after filtering
  if (validNumbers.length === 0) {
    return 0;
  }

  // 4. Sum up the valid numbers using .reduce()
  const totalSum = validNumbers.reduce((accumulator, current) => accumulator + current, 0);

  // 5. Return the calculated average
  return totalSum / validNumbers.length;
}
What Changed & Why?
Array.isArray(arr) Check: The original function would crash with a TypeError if you accidentally passed a string, an object, or null. This guard clause ensures it only proceeds if it receives an actual array.

Empty Array Handling (arr.length === 0): In your original code, an empty array would result in 0 / 0, which returns NaN (Not a Number) in JavaScript. Now, it explicitly returns 0.

.filter() for Non-Numbers: If your input array contains strings or booleans (e.g., [10, "apple", 20]), the original code would perform string concatenation or math errors, resulting in NaN. The .filter() method creates a clean array of only valid numbers.

.reduce() instead of a for loop: Using .reduce() is the modern JavaScript standard for accumulating values. It is more declarative and easier to read at a glance than managing loop counters (let i = 0).

3) Explain prompt engineering

Prompt design 
Prompt engineering is the practice of structuring, designing, and refining text inputs (prompts) to get the most accurate, relevant, and useful responses out of Generative AI models like ChatGPT, Gemini, or Claude.

Think of a Large Language Model (LLM) as an incredibly brilliant, well-read intern who has access to almost all human knowledge, but lacks mind-reading skills. If you give them a vague, lazy assignment, you’ll get a vague, lazy result. Prompt engineering is simply the art of giving clear, contextual instructions so the intern hits a home run on the first try.

The Core Components of a Perfect Prompt
You don't need to write a massive wall of text every time you talk to an AI. However, when you need high-quality results for complex tasks, a well-engineered prompt typically blends these four elements:

Role/Persona (Who is the AI?): Tell the AI what perspective it should take.

Example: "Act as a senior cybersecurity auditor..." or "Speak like a humorous middle school science teacher."

Context (What is the background?): Give the AI the surrounding details, target audience, or constraints.

Example: "...explaining data privacy to a non-technical marketing team of 5 people."

Task/Instruction (What do you want it to do?): Use clear, action-oriented verbs.

Example: "Analyze this text and extract the top 3 security vulnerabilities."

Output Indicator (How should it look?): Define the formatting, tone, or length restrictions.

Example: "Format the output as a Markdown table with columns for 'Vulnerability', 'Risk Level', and 'Fix'."

4 Essential Prompt Engineering Techniques
As the field has evolved, researchers have developed specific framework techniques to handle advanced reasoning tasks.

1. Few-Shot Prompting
Instead of just telling the AI what you want, you show it by providing a few examples of input and output. This is incredibly useful for matching a specific tone, formatting, or classification style.

Prompt: > "Great customer service, loved it!" -> Positive
"The shipping took three weeks and it arrived broken." -> Negative
"It's an okay product, does what it says." -> Neutral
"I hate how loud the buttons click." -> [AI will automatically answer "Negative"]

2. Chain-of-Thought (CoT) Prompting
For complex logic, math, or troubleshooting, you explicitly ask the AI to "think step-by-step" before blurting out an answer. Forcing the model to write out its reasoning path drastically reduces errors (hallucinations).

3. Role-Based Prompting
Assigning a specific domain expertise to the model adjusts its vocabulary, depth of explanation, and systemic assumptions, immediately narrowing down the scope of its database to the most relevant information.

4. Retrieval-Augmented Generation (RAG)
While not strictly a text-only prompt trick, this involves feeding specific documentation, articles, or code directly into the prompt window alongside your question: "Based on the attached PDF manual, how do I reset the server?" This anchors the AI to real, verified facts.

Why It Matters (It's Not Just "Talking to Chatbots")
Prompt engineering has shifted from a tech hobby to a foundational modern skill. It bridges the gap between human intent and machine execution.

Saves Time and Money: Well-engineered prompts reduce the number of iterations required to get a usable output, lowering API computing costs and user frustration.

Unlocks Hidden Capabilities: LLMs are capable of highly nuanced logic, translation, coding, and creative problem-solving that generic prompts ("write an essay about X") completely fail to trigger.

Foundation for AI Agents: When developers build automated AI apps, they must hard-code highly robust, bulletproof prompts into the background to ensure the AI behaves predictably every single time.

4) Refactor a Node.js Express middleware rate limiter

Node.js 
Here is a clean, dependency-free Express middleware function that handles rate limiting using an in-memory store.

Since it uses standard JavaScript objects, it's perfect for a single-instance server. (Note: For production apps scaled across multiple servers, you'd eventually want to swap the in-memory object for something like Redis).

JavaScript
/**
 * Simple In-Memory Rate Limiting Middleware
 * Limits requests to 100 per 15 minutes per IP address.
 */
function rateLimiter(req, res, next) {
    // 1. Define the limits
    const WINDOW_MS = 15 * 60 * 1000; // 15 minutes in milliseconds
    const MAX_REQUESTS = 100;

    // 2. Initialize the in-memory store if it doesn't exist on the global/app level
    // We attach it to `req.app.locals` so it persists across requests without polluting global scope
    if (!req.app.locals.rateLimitStore) {
        req.app.locals.rateLimitStore = new Map();
    }
    const store = req.app.locals.rateLimitStore;

    // 3. Identify the user by their IP address
    // 'req.ip' works well, but if you're behind a proxy (like Nginx or Heroku), 
    // make sure to enable 'trust proxy' in your Express app settings.
    const ip = req.ip;
    const now = Date.now();

    // 4. Clean up expired entries periodically to prevent memory leaks
    // (Optional but highly recommended for an in-memory solution)
    for (const [storedIp, data] of store.entries()) {
        if (now - data.resetTime > WINDOW_MS) {
            store.delete(storedIp);
        }
    }

    // 5. Check if the IP already has a record
    if (!store.has(ip)) {
        // First request from this IP in the current window
        store.set(ip, {
            count: 1,
            resetTime: now + WINDOW_MS
        });

        // Set standard rate-limiting headers
        res.setHeader('X-RateLimit-Limit', MAX_REQUESTS);
        res.setHeader('X-RateLimit-Remaining', MAX_REQUESTS - 1);
        res.setHeader('X-RateLimit-Reset', Math.ceil((now + WINDOW_MS) / 1000));

        return next();
    }

    const record = store.get(ip);

    // 6. Check if the 15-minute window has passed for this specific IP
    if (now > record.resetTime) {
        // Window expired, reset the tracking for this IP
        record.count = 1;
        record.resetTime = now + WINDOW_MS;

        res.setHeader('X-RateLimit-Limit', MAX_REQUESTS);
        res.setHeader('X-RateLimit-Remaining', MAX_REQUESTS - 1);
        res.setHeader('X-RateLimit-Reset', Math.ceil(record.resetTime / 1000));

        return next();
    }

    // 7. Increment the request count
    record.count++;

    // Calculate remaining requests and reset time in seconds
    const remaining = Math.max(0, MAX_REQUESTS - record.count);
    const resetTimeSeconds = Math.ceil(record.resetTime / 1000);

    res.setHeader('X-RateLimit-Limit', MAX_REQUESTS);
    res.setHeader('X-RateLimit-Remaining', remaining);
    res.setHeader('X-RateLimit-Reset', resetTimeSeconds);

    // 8. Block the request if they exceed the limit
    if (record.count > MAX_REQUESTS) {
        res.setHeader('Retry-After', Math.ceil((record.resetTime - now) / 1000));
        return res.status(429).json({
            error: 'Too many requests',
            message: 'You have exceeded the 100 requests per 15 minutes limit. Please try again later.'
        });
    }

    // 9. If under the limit, proceed to the next middleware/route handler
    next();
}

module.exports = rateLimiter;

How to use it in your Express app
You can apply this globally to all routes, or selectively to sensitive endpoints like login or API routes.

JavaScript
const express = require('express');
const rateLimiter = require('./rateLimiter'); // Path to the middleware file

const app = express();

// If you are behind a proxy (Nginx, Apache, Cloudflare, Heroku, etc.)
// app.set('trust proxy', 1); 

// Apply the rate limiter globally
app.use(rateLimiter);

app.get('/api/data', (req, res) => {
    res.json({ message: "Hello World! Connection successful." });
});

app.listen(3000, () => console.log('Server running on port 3000'));
Key Highlights of this Implementation:
Map Object: We use a JavaScript Map instead of a plain object because it performs better with frequent additions and deletions of keys (IP addresses).

Standard HTTP Headers: It includes X-RateLimit-* headers to nicely communicate the limits to the client, along with a Retry-After header when a 429 status is triggered.

Memory Management: It includes a lightweight cleanup loop that deletes stale IP entries so your server's RAM doesn't grow indefinitely over time.

5) Explain why useEffect causes an infinite render loop

React 
This React useEffect hook causes an infinite render loop:



useEffect(() => {

  setCount(count + 1);

}, [count]);



Explain why this happens and show the corrected version.

The infinite loop happens because of a cyclical dependency between your useEffect hook and the component's state.

Here is the exact step-by-step breakdown of why this code causes a crash:

Initial Render: The component renders for the first time with the initial count value (e.g., 0).

Effect Runs: Because the component just rendered, the useEffect hook triggers.

State Updates: Inside the effect, setCount(count + 1) is called, changing the state to 1.

Re-render Triggered: In React, changing a state variable automatically forces the component to re-render.

Dependency Check: After the re-render, React looks at the useEffect dependency array [count]. It notices that count has changed from 0 to 1.

Effect Runs Again: Because the dependency changed, the effect runs a second time and calls setCount(1 + 1).

The Loop Continues: This triggers another re-render, which triggers the effect, which updates the state, creating an infinite loop that will quickly crash your browser or hit React's maximum update depth limit.

The Fix
How you fix this depends entirely on what you are actually trying to achieve.

Scenario A: You just wanted to update the count based on its previous value
If you don't actually need the effect to trigger every time count changes, and you just want to safely update the state, you should use a functional state update. This allows you to remove count from the dependency array entirely.

JavaScript
useEffect(() => {
  // Pass a function to setCount. 
  // 'prevCount' is guaranteed to be the most up-to-date state.
  setCount(prevCount => prevCount + 1);
}, []); // Empty array means this runs ONLY ONCE when the component mounts

Scenario B: You want to run code when count changes
If your actual goal is to trigger a side effect (like fetching data or logging) whenever count changes, do not update count inside that same effect.

JavaScript
useEffect(() => {
  // Do something else when count changes, but DO NOT set the count here.
  console.log(`The count is now ${count}`);
}, [count]); // Safely tracks count without looping
